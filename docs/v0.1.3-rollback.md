# v0.1.3 Schema Enforcement — Rollback Contract

> **Purpose**: Document v0.1.3 behavior change, kill-switch knob, and rollback procedure if v0.1.3 schema enforcement causes unexpected operational impact.

**Version**: v0.1.3
**Released**: 2026-05-02
**Pilot evidence**: BeiliSystem PR #30/#34 dogfood (3 stages, 25 dispatches, 3 incomplete returns observed)

---

## What v0.1.3 changed

Pre-v0.1.3, dispatch-header.md described 11 mandatory return-contract fields and `stage-runbook.md` §EXECUTING step 5 referenced "validate all 11 required fields", but:

1. **No persistent record** — audit-trail.jsonl did not log validation outcome; INCOMPLETE handling was procedural-only
2. **Retry budget ambiguous** — dispatch-header.md §Return contract said "re-dispatched once" without clarifying whether the retry consumed the §EXECUTING step 7 step-review retry slot
3. **Edge case missing** — second INCOMPLETE was procedurally ESCALATED but not separately recorded in audit-trail with diagnostic `schema_validation_status`

v0.1.3 closes these gaps by:

1. **Adding `schema_validation_status` field** to audit-trail.jsonl row schema (enum: `pass | rejected_and_retried | rejected_and_escalated | null`)
2. **Adding explicit step 5 validation flow** in `stage-runbook.md` §EXECUTING with the 11-field canonical list inline (no longer requiring TeamLead to cross-load anti-rubber-stamp.md)
3. **Adding retry-pool separation note** in `dispatch-header.md` §Return contract — schema-validation re-dispatch has its OWN 1-retry pool, distinct from step-review retry pool
4. **Adding worked examples** (`pass` / `rejected_and_retried` / `rejected_and_escalated`) for TeamLead reference at runtime

## What v0.1.3 did NOT change

- The 11 mandatory fields list itself (still per `anti-rubber-stamp.md` §Mandatory rejection criteria)
- §EXECUTING step 7 step-review retry pool logic (1 retry per cadence rule, second FAIL → ESCALATED)
- §EXECUTING step 7a KMR per-task proxy logic
- Mini Gate retry pool logic (Branch A vs Branch B routing per `three-gates.md`)
- Any `meta` block field semantics (Phase 2 RSS)
- Any trust_tier / Rule 0 / Rule 0.5 / Rule 2 logic (Phase 2/3 N1)

## Kill-switch knob

`schema_enforcement_mode` knob added to `templates/budget-proposal.md.tpl`:

| Mode | Behavior | Activation |
|---|---|---|
| `strict` (default) | Full v0.1.3 enforcement: validate → INCOMPLETE → re-dispatch → second INCOMPLETE → ESCALATED. Records `schema_validation_status` ∈ {`pass`, `rejected_and_retried`, `rejected_and_escalated`}. | Default — no consent needed |
| `warn` | Validate but DO NOT reject; logs `schema_validation_status` to audit-trail; v0.1.2 silent-acceptance preserved. Use this if v0.1.3 ESCALATED is too aggressive for the project context. | **CCB-Heavy** (rollback of deployed enforcement) |
| `off` | Skip validation entirely; `schema_validation_status: null` for all dispatches. Effectively pre-v0.1.3 behavior. | **CCB-Heavy** (rollback of deployed enforcement) |

## Rollback procedures

### Scenario 1 — Project-level rollback (one project misbehaves under v0.1.3)

1. Open CCB-Heavy at next CEO_Gate per `references/pmp-ccb.md`
2. Document evidence (e.g., "PM persistently emits 10/11 fields; ESCALATED 3× in Stage 2 blocks completion")
3. CEO approves `schema_enforcement_mode` change `strict → warn` (or `off`)
4. Update PROGRESS.md `## Budget Baseline` knob row
5. TeamLead resumes per new mode; remaining dispatches log `schema_validation_status: null` (off) or still validate but accept (warn)

### Scenario 2 — Plugin-level rollback (v0.1.3 broken across all projects)

1. **Symptom check**: query existing projects' audit-trail.jsonl
   ```bash
   jq -s '[.[].schema_validation_status] | group_by(.) | map({val: .[0], n: length})' audit-trail.jsonl
   ```
   If `rejected_and_escalated` count > expected (per `stage-runbook.md` §Schema validation worked examples Anti-anchoring note: should be near-zero), suspect plugin-level defect
2. **Surgical revert** preferred over full plugin downgrade:
   - Surface as plugin issue under `~/.claude/projects/-Users-HsuTse/memory/issues/`
   - Apply hotfix patch to v0.1.3 (release v0.1.4 with fix)
3. **Full downgrade** (last resort):
   ```bash
   cd /Users/HsuTse/.claude/plugins/teamwork-leader
   git checkout v0.1.2
   ```
   - Existing projects still have `schema_validation_status` field in audit-trail (reading is forward-compatible: legacy `null` is documented behavior)
   - **Do NOT** rewrite historical audit-trail rows — they are append-only project artifacts

## Migration / forward-compat

- v0.1.2 audit-trail.jsonl rows lack `schema_validation_status` field. Tooling treats absent field as `null` (per progress-md-schema.md §`schema_validation_status` legacy clause).
- v0.1.3 reading v0.1.2 row → `null`; v0.1.2 reading v0.1.3 row → unknown extra field, structurally harmless (jq skips, JSON.parse ignores)
- No retroactive backfill of v0.1.2 rows
- New projects post-v0.1.3 ship → all rows have `schema_validation_status` non-null (unless `schema_enforcement_mode == off`)
